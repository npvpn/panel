# Роли нод и каскад (entry / exit / direct)

Как настраивать цепочку «входная → выходная» и обычные ноды через панель, что писать в
Core settings (`xray_config.json`), а что панель генерирует сама.

## Главная идея

Сырой xray-конфиг каскада **руками больше не пишется**. На карточке каждой ноды есть
**роль**, а каскадные куски конфига панель добавляет автоматически при пуше конфига на ноду:

| Роль | Что панель делает с конфигом ноды |
|------|-----------------------------------|
| **Direct** (по умолчанию) | ничего не добавляет — нода работает как раньше |
| **Exit** (выходная) | добавляет служебный принимающий инбаунд `CASCADE_IN` (порт **2096**, vless+TCP+REALITY) |
| **Entry** (входная) | на каждый маршрут добавляет `outbound` на выходную + routing-правило по инбаунду |

Роль **аддитивна**: она только *добавляет* каскад поверх обычного обслуживания юзеров,
ничего не отключает (см. «Выходная + direct одновременно»).

Порядок сборки конфига ноды:

```
Core settings (xray_config.json: инбаунды юзеров, базовые outbounds/routing)
   → фильтр по инбаундам ноды (назначение инбаундов на карточке ноды)
   → + каскад по роли (CASCADE_IN на exit / CASCADE_OUT_* + routing на entry)
   → пуш на ноду
```

Главный core (master) каскадом не дополняется и поднимает всё как раньше.

---

## Core settings (`xray_config.json`) — что писать

Это **общий каталог**, который раздаётся всем нодам (каждой — её подмножество инбаундов).

**Писать сюда нужно:**

- **Пользовательские инбаунды** — то, к чему коннектятся юзеры (vless/vmess/trojan/ss,
  REALITY, xhttp, grpc и т.д.). Это и есть «страны/транспорты», которые потом назначаются
  на ноды и выбираются в подписке.
- **Базовые outbounds**: `DIRECT` (freedom), `BLOCK` (blackhole), при желании `ADS`.
- **Базовые routing-правила**: блок bittorrent/приватных IP/гео и т.п.

**Писать сюда НЕ нужно (бессмысленно — панель сгенерит сама):**

- ❌ Принимающий каскадный инбаунд выходной ноды (`CASCADE_IN`, порт 2096). Генерится при
  роли Exit, в Core settings его нет и быть не должно.
- ❌ Каскадные outbounds (`CASCADE_OUT_<id>`) на входной ноде. Генерятся из маршрутов.
- ❌ Routing-правила каскада (`inboundTag → CASCADE_OUT_*`). Генерятся из маршрутов.

Если раньше каскад был прописан в `xray_config.json` руками (отдельный `outbound` вида
`OUT_TO_*` + routing-правило по его tag) — после перевода на роли эти ручные куски нужно
**удалить** (см. «Миграция ручного каскада»).

> Важно: инбаунд, на который маршрутизирует входная нода, **должен существовать в Core
> settings** как обычный пользовательский инбаунд и быть назначенным входной ноде (иначе
> фильтр инбаундов ноды его вырежет, и routing-правило сошлётся на несуществующий инбаунд).

## Настройка выходной ноды (Exit)

1. **Nodes** → открыть карточку зарубежной ноды.
2. **Node role** → **Exit (Выходная)** → Save.
3. **Reconnect** ноды.

Что произойдёт: панель один раз сгенерирует `cascade_params` ноды (REALITY-keypair, uuid,
SNI `xapi.ozon.ru`, порт 2096) и при пуше добавит ей служебный инбаунд `CASCADE_IN`. Этот
инбаунд в UI не показывается и в `xray_config.json` не пишется.

Проверка на ноде:

```bash
ss -tlpn | grep 2096
```

Должен слушаться `0.0.0.0:2096` — **в дополнение** к обычным инбаундам ноды.

Больше на выходной ничего настраивать не нужно.

---

## Настройка входной ноды (Entry)

1. **Nodes** → открыть карточку входной (напр. РУ) ноды.
2. **Node role** → **Entry (Входная)**. Появится блок **«Cascade routes»**.
3. **Add route** → в строке выбрать:
   - **инбаунд** — тот, к которому коннектится юзер (должен быть назначен этой ноде);
   - **выходную ноду** — из списка (туда попадают только ноды с ролью Exit).
   - Несколько маршрутов = 1 входная → N выходных (каждый свой инбаунд → своя выходная).
4. Save → **Reconnect** входной ноды.

Что произойдёт: на входной добавится `outbound` на каждую выбранную выходную (на её
`CASCADE_IN:2096`) и routing-правило `inboundTag(выбранный инбаунд) → этот outbound`.

**Как юзер выбирает страну выхода:** в подписке хост, указывающий на выбранный инбаунд
входной ноды, = трафик пойдёт через привязанную к этому инбаунду выходную. То есть «страна
выхода» определяется тем, какой инбаунд входной ноды юзер включил.

Проверка сквозного трафика (подключиться клиентом к нужному инбаунду входной):

```bash
curl -s https://api.ipify.org
```

Должен вернуться IP **выходной** ноды.

## Прямые ноды (Direct) — по умолчанию

Ничего делать не нужно. Роль `Direct` стоит по умолчанию, существующие ноды её и получают.
Direct-нода работает ровно как раньше (никакого каскада), обслуживает назначенные ей
инбаунды (или все, если инбаунды не назначены — поведение по умолчанию).

---

## Выходная + direct одновременно

Отдельной «комбинированной» роли не нужно. **Роль Exit уже включает обычное обслуживание
юзеров.** Выходная нода одновременно:

- принимает каскад от входных (`CASCADE_IN`), и
- отдаёт свои пользовательские инбаунды напрямую (как direct).

Что выбирать, если хочешь и выход каскада, и прямых юзеров на одной ноде:

- **Role = Exit.**
- В мультиселекте инбаундов на карточке ноды:
  - назначить **конкретные** инбаунды → нода отдаёт юзерам только их (+ принимает каскад);
  - оставить **пустым** → нода отдаёт юзерам **все** инбаунды каталога (+ принимает каскад).

`CASCADE_IN` висит на отдельном порту (2096) и обычным юзерам не мешает — они подключаются к
своим инбаундам, egress у всех общий (DIRECT в интернет). `Direct` нужен **только** когда
каскад на ноде не нужен вообще.

## Миграция ручного РУ-каскада

Если каскад сейчас прописан руками в `xray_config.json` (отдельный `outbound` + routing по
его tag):

1. Зарубежную ноду → роль **Exit**, Reconnect (сгенерятся `cascade_params`).
2. РУ-ноду → роль **Entry**, добавить маршрут `(РУ-инбаунд → зарубежная нода)`, Reconnect.
3. Проверить сквозной трафик (`curl` через РУ-инбаунд = IP зарубежной).
4. **Только после успешной проверки** — удалить из `xray_config.json` ручные каскадные
   `outbounds` (`OUT_TO_*`) и их routing-правила (оставить статические DIRECT/BLOCK/ADS и
   базовые блок-правила).
5. Reconnect, повторно проверить трафик.

## Заметки / troubleshooting

- После любого изменения роли/маршрутов нужен **Reconnect** ноды (пуш нового конфига).
- Дефолты каскада: порт `2096`, SNI/dest `xapi.ozon.ru:443`, `flow: xtls-rprx-vision`,
  `fingerprint: chrome`. Заданы в `app/xray/cascade_keys.py` / `app/xray/cascade_config.py`.
- `cascade_params` (с приватным REALITY-ключом) хранятся в БД на ноде и **не отдаются** в
  API-ответах и подписках.
- Если выходная ещё без `cascade_params` (роль не применилась/не было Reconnect), маршрут на
  неё во входном конфиге **пропускается** (без ошибки) — сначала подними выходную.
- REALITY-ключи генерятся через `xray x25519`. На xray 26.x формат вывода — `PrivateKey:` /
  `Password (PublicKey):` (парсер `app/xray/core.py get_x25519` понимает и старый, и новый).
