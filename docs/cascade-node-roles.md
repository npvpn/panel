# Роли нод и каскад (entry / exit / direct)

Как настраивать цепочку «входная → выходная» и обычные ноды через панель, что писать в
Core settings (`xray_config.json`), а что панель добавляет сама.

## Главная идея

Сырой xray-конфиг каскада **руками больше не пишется**. На карточке каждой ноды есть
**роль**, а каскадные куски конфига панель добавляет автоматически при пуше конфига на ноду.
Принимающая точка каскада на выходной — это **обычный VLESS+TCP+REALITY инбаунд из каталога**
`xray_config.json` (тот, что настраивается в Core settings), а не служебный сгенерённый инбаунд.

| Роль | Что панель делает с конфигом ноды |
|------|-----------------------------------|
| **Direct** (по умолчанию) | ничего не добавляет — нода работает как раньше |
| **Exit** (выходная) | в выбранный каталожный VLESS+TCP+REALITY инбаунд дописывает служебного cascade-клиента (uuid) |
| **Entry** (входная) | на каждый маршрут добавляет `outbound` на cascade-инбаунд выходной + routing-правило по инбаунду |

Роль **аддитивна**: она только *добавляет* каскад поверх обычного обслуживания юзеров,
ничего не отключает (см. «Выходная + direct одновременно»).

Порядок сборки конфига ноды:

```text
Core settings (xray_config.json: инбаунды юзеров, базовые outbounds/routing)
   → фильтр по инбаундам ноды (назначение инбаундов на карточке ноды)
   → + каскад по роли (служебный client в инбаунд на exit / outbound+routing на entry)
   → пуш на ноду
```

Главный core (master) каскадом не дополняется и поднимает всё как раньше.

---

## Core settings (`xray_config.json`) — что писать

Это **общий каталог**, который раздаётся всем нодам (каждой — её подмножество инбаундов).

**Писать сюда нужно:**

- **Пользовательские инбаунды** — то, к чему коннектятся юзеры (vless/vmess/trojan/ss,
  REALITY, xhttp, grpc и т.д.). Это и есть «страны/транспорты», которые назначаются на ноды
  и выбираются в подписке.
- **Cascade-инбаунд выходной** — это **обычный VLESS+TCP+REALITY инбаунд** в Core settings.
  Его нужно завести (или использовать существующий), назначить выходной ноде и выбрать в
  маршруте на входной. Панель сама допишет в него служебного cascade-клиента — отдельный
  инбаунд для каскада создавать не надо.
- **Базовые outbounds**: `DIRECT` (freedom), `BLOCK` (blackhole), при желании `ADS`.
- **Базовые routing-правила**: блок bittorrent/приватных IP/гео и т.п.

**Писать сюда НЕ нужно (панель сгенерит сама):**

- ❌ Каскадные outbounds (`CASCADE_OUT_*`) на входной ноде — генерятся из маршрутов.
- ❌ Routing-правила каскада (`inboundTag → CASCADE_OUT_*`) — генерятся из маршрутов.
- ❌ Служебный cascade-client в инбаунде выходной — панель добавит его сама по роли Exit.

Если раньше каскад был прописан в `xray_config.json` руками (отдельный `outbound` вида
`OUT_TO_*` + routing-правило по его tag) — после перевода на роли эти ручные куски нужно
**удалить** (см. «Миграция ручного каскада»).

> Важно: cascade-инбаунд выходной и инбаунд, на который маршрутизирует входная нода,
> **должны существовать в Core settings** как обычные пользовательские инбаунды и быть
> назначены соответствующим нодам (иначе фильтр инбаундов ноды их вырежет).

## Настройка выходной ноды (Exit)

1. В Core settings завести (или выбрать существующий) **VLESS+TCP+REALITY** инбаунд для
   каскада.
2. **Nodes** → открыть карточку зарубежной ноды → назначить ей этот инбаунд (мультиселект
   инбаундов).
3. **Node role** → **Exit (Выходная)** → Save.
4. **Reconnect** ноды.

Что произойдёт: панель один раз сгенерирует служебный uuid выходной (`cascade_params={"uuid"}`)
и при пуше допишет служебного cascade-клиента в назначенные ей cascade-инбаунды (те, что
выбраны в маршрутах входных нод). Отдельный порт/инбаунд не создаётся — каскад принимается
тем же каталожным инбаундом.

Проверка на ноде (порт **каталожного** cascade-инбаунда, напр. 8443):

```bash
ss -tlpn | grep <порт cascade-инбаунда>
```

Должен слушаться этот порт — он же обслуживает обычных юзеров инбаунда.

## Настройка входной ноды (Entry)

1. **Nodes** → открыть карточку входной (напр. РУ) ноды.
2. **Node role** → **Entry (Входная)**. Появится блок **«Cascade routes»**.
3. **Add route** → в строке выбрать **три** поля:
   - **entry-инбаунд** — тот, к которому коннектится юзер (должен быть назначен этой ноде);
   - **выходную ноду** — из списка (туда попадают только ноды с ролью Exit);
   - **cascade-инбаунд** — инбаунд на выходной, через который строить связку (VLESS+TCP+REALITY).
   - Несколько маршрутов = 1 входная → N выходных (каждый свой entry-инбаунд → своя выходная).
4. Save → **Reconnect** входной ноды.

Что произойдёт: на входной добавится `outbound` на cascade-инбаунд выбранной выходной
(адрес/порт/REALITY-параметры — из определения этого инбаунда; publicKey деривится из его
privateKey) и routing-правило `entry-инбаунд → этот outbound`.

**Как юзер выбирает страну выхода:** в подписке хост, указывающий на выбранный entry-инбаунд
входной ноды, = трафик пойдёт через привязанную к этому инбаунду выходную. «Страна выхода»
определяется тем, какой entry-инбаунд входной ноды юзер включил.

Проверка сквозного трафика (подключиться клиентом к нужному entry-инбаунду входной):

```bash
curl -s https://api.ipify.org
```

Должен вернуться IP **выходной** ноды.

## Прямые ноды (Direct) — по умолчанию

Ничего делать не нужно. Роль `Direct` стоит по умолчанию, существующие ноды её и получают.
Direct-нода работает ровно как раньше (никакого каскада), обслуживает назначенные ей
инбаунды (или все, если инбаунды не назначены — поведение по умолчанию).

---

## Выходная + direct одновременно

Отдельной «комбинированной» роли не нужно. **Роль Exit уже включает обычное обслуживание
юзеров.** Выходная нода одновременно:

- принимает каскад от входных (служебный client в cascade-инбаунде), и
- отдаёт свои пользовательские инбаунды напрямую (как direct).

Что выбирать, если хочешь и выход каскада, и прямых юзеров на одной ноде:

- **Role = Exit.**
- В мультиселекте инбаундов на карточке ноды:
  - назначить **конкретные** инбаунды (включая cascade-инбаунд) → нода отдаёт юзерам только их;
  - оставить **пустым** → нода отдаёт юзерам **все** инбаунды каталога.

Служебный cascade-client живёт в обычном инбаунде с отдельным uuid и обычным юзерам не мешает
— они подключаются своими credentials, egress у всех общий (DIRECT в интернет). `Direct`
нужен **только** когда каскад на ноде не нужен вообще.

## Миграция ручного РУ-каскада

Если каскад сейчас прописан руками в `xray_config.json` (отдельный `outbound` + routing по
его tag):

1. На зарубежной ноде завести/назначить VLESS+TCP+REALITY инбаунд для каскада, роль **Exit**,
   Reconnect (сгенерится служебный uuid, в инбаунд допишется cascade-client).
2. РУ-ноду → роль **Entry**, добавить маршрут `(РУ-инбаунд → зарубежная нода → cascade-инбаунд)`,
   Reconnect.
3. Проверить сквозной трафик (`curl` через РУ-инбаунд = IP зарубежной).
4. **Только после успешной проверки** — удалить из `xray_config.json` ручные каскадные
   `outbounds` (`OUT_TO_*`) и их routing-правила (оставить статические DIRECT/BLOCK/ADS и
   базовые блок-правила).
5. Reconnect, повторно проверить трафик.

## Заметки / troubleshooting

- После любого изменения роли/маршрутов нужен **Reconnect** ноды (пуш нового конфига).
- Дефолты каскад-outbound: `flow: xtls-rprx-vision`, `fingerprint: chrome` (заданы в
  `app/xray/cascade_config.py`). Порт, SNI (`serverName`), `shortId`, `publicKey` берутся из
  определения cascade-инбаунда в `xray_config.json` (publicKey деривится из его privateKey).
- Служебный uuid выходной хранится в `Node.cascade_params={"uuid"}` и **не отдаётся** в
  API-ответах и подписках.
- Если выходная ещё без служебного uuid (роль не применилась/не было Reconnect) или
  cascade-инбаунд не найден в каталоге — маршрут на неё во входном конфиге **пропускается**
  (без ошибки) — сначала подними выходную и заведи инбаунд.
- REALITY-ключи парсятся через `xray x25519`. На xray 26.x формат вывода — `PrivateKey:` /
  `Password (PublicKey):` (парсер `app/xray/core.py get_x25519` понимает и старый, и новый).
